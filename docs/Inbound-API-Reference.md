# Greet — Inbound API Reference

**Version:** 1.0.0 | **Date:** April 2026
**Base Path:** `/api/v1/inbound`
**Auth:** Bearer token (`Authorization: Bearer grt_...`)
**Rate Limit:** 60 requests/minute per API key
**Max Payload:** 1 MB

---

## Authentication

All inbound endpoints require an API key generated by a Greet super admin. The key is passed as a Bearer token:

```
Authorization: Bearer grt_abc123...
```

API keys are:
- Scoped to a single customer account
- Hashed (SHA-256) at rest — the raw key is only shown once at creation
- Optionally restricted to an IP allowlist
- Revocable without affecting other keys
- Logged in the audit trail on every use

The `inbound_api` feature flag must be enabled for the customer account.

## Endpoints

### POST `/api/v1/inbound/events`

Create or update an event. Matched by `eventCode` (+ optional `accountCode`). Updates never overwrite local configuration (badges, workflow, staff settings, etc.).

**Request:**

```json
{
  "eventCode": "EVT-2026-ANNUAL",
  "accountCode": "ACCT-001",
  "name": "Annual Developer Conference 2026",
  "eventDate": "2026-06-15T09:00:00Z",
  "startDate": "2026-06-15T09:00:00Z",
  "endDate": "2026-06-17T17:00:00Z",
  "timezone": "America/Los_Angeles",
  "location": "Moscone Center",
  "venue": "Hall B"
}
```

| Field | Type | Required | Notes |
|-------|------|----------|-------|
| eventCode | string | Yes | Unique identifier from source system |
| accountCode | string | No | Narrows event lookup when codes aren't globally unique |
| name | string | Yes | Max 500 chars |
| eventDate | ISO 8601 datetime | Yes | Primary event date |
| startDate | ISO 8601 datetime | No | Event start (if multi-day) |
| endDate | ISO 8601 datetime | No | Event end |
| timezone | string | No | IANA timezone (e.g., `America/Los_Angeles`) |
| location | string | No | Max 500 chars |
| venue | string | No | Max 500 chars |

**Response — Created (201):**

```json
{
  "success": true,
  "action": "created",
  "event": {
    "id": "evt_abc123",
    "name": "Annual Developer Conference 2026",
    "eventCode": "EVT-2026-ANNUAL",
    "eventDate": "2026-06-15T09:00:00.000Z",
    "..."
  }
}
```

**Response — Updated (200):**

```json
{
  "success": true,
  "action": "updated",
  "event": {
    "id": "evt_abc123",
    "name": "Annual Developer Conference 2026",
    "..."
  }
}
```

### POST `/api/v1/inbound/attendees`

Batch create or update attendees for an event. Matched by `externalId` + `eventId`. Updates never overwrite check-in status, badge print status, or workflow data.

**Max batch size:** 100 attendees per request.

**Request:**

```json
{
  "eventCode": "EVT-2026-ANNUAL",
  "accountCode": "ACCT-001",
  "attendees": [
    {
      "externalId": "REG-10042",
      "firstName": "Jane",
      "lastName": "Smith",
      "email": "jane@example.com",
      "company": "Acme Corp",
      "title": "VP Engineering",
      "participantType": "VIP",
      "registrationStatus": "Confirmed",
      "registrationStatusLabel": "Confirmed Attendee",
      "orderCode": "ORD-5001",
      "externalProfileId": "PROF-882",
      "phone": "+14155551234",
      "customFields": {
        "dietary": "Vegetarian",
        "tshirt": "L"
      }
    }
  ]
}
```

| Field | Type | Required | Notes |
|-------|------|----------|-------|
| eventCode | string | Yes | Must match an existing event |
| accountCode | string | No | Narrows event lookup |
| attendees | array | Yes | 1–100 items |
| attendees[].externalId | string | Yes | Unique ID from source system |
| attendees[].firstName | string | Yes | Max 200 chars |
| attendees[].lastName | string | Yes | Max 200 chars |
| attendees[].email | string | Yes | Valid email |
| attendees[].company | string | No | Max 200 chars |
| attendees[].title | string | No | Max 200 chars |
| attendees[].participantType | string | No | Default: `"General"`. Max 100 chars |
| attendees[].registrationStatus | string | No | Default: `"Registered"` on create; unchanged on update |
| attendees[].registrationStatusLabel | string | No | Display label for status |
| attendees[].orderCode | string | No | Group/order identifier for group check-in |
| attendees[].externalProfileId | string | No | Profile ID for QR code encoding |
| attendees[].phone | string | No | Max 50 chars |
| attendees[].customFields | object | No | Key-value pairs (string values only) |

**Response (200):**

```json
{
  "success": true,
  "processed": 10,
  "created": 7,
  "updated": 3,
  "errors": []
}
```

**Response with partial failures (200):**

```json
{
  "success": true,
  "processed": 8,
  "created": 5,
  "updated": 3,
  "errors": [
    {
      "index": 4,
      "externalId": "REG-BAD",
      "error": "Duplicate email"
    },
    {
      "index": 9,
      "externalId": "REG-ALSO-BAD",
      "error": "Processing failed"
    }
  ]
}
```

> **Note:** Partial failures do not roll back successful records. The `errors` array identifies which items failed and why. The `index` field corresponds to the item's position in the request array.

### POST `/api/v1/inbound/session-registrations`

Batch register attendees for sessions. Both the attendee and session must already exist in the event (matched by their `externalId` values).

**Max batch size:** 500 registrations per request.

**Request:**

```json
{
  "eventCode": "EVT-2026-ANNUAL",
  "accountCode": "ACCT-001",
  "registrations": [
    {
      "attendeeExternalId": "REG-10042",
      "sessionExternalId": "SESS-301",
      "status": "registered"
    },
    {
      "attendeeExternalId": "REG-10043",
      "sessionExternalId": "SESS-301",
      "status": "waitlisted"
    }
  ]
}
```

| Field | Type | Required | Notes |
|-------|------|----------|-------|
| eventCode | string | Yes | Must match an existing event |
| accountCode | string | No | Narrows event lookup |
| registrations | array | Yes | 1–500 items |
| registrations[].attendeeExternalId | string | Yes | Must match existing attendee |
| registrations[].sessionExternalId | string | Yes | Must match existing session |
| registrations[].status | enum | No | `"registered"` (default), `"waitlisted"`, or `"cancelled"` |

**Response (200):**

```json
{
  "success": true,
  "processed": 2,
  "created": 2,
  "updated": 0,
  "errors": []
}
```

**Response with lookup failures (200):**

```json
{
  "success": true,
  "processed": 1,
  "created": 1,
  "updated": 0,
  "errors": [
    {
      "index": 1,
      "error": "Attendee not found: REG-UNKNOWN"
    }
  ]
}
```

### GET `/api/v1/inbound/attended`

Retrieve all checked-in attendees for an event, including profile data, workflow question responses, and disclaimer signatures.

**Query Parameters:**

| Parameter | Required | Default | Description |
|-----------|----------|---------|-------------|
| eventCode | Yes | — | Event identifier from source system |
| accountCode | No | — | Narrows event lookup |
| limit | No | 100 | Records per page (1–500 max) |
| offset | No | 0 | Skip this many records (for pagination) |
| since | No | — | ISO 8601 datetime — only check-ins **after** this time |
| before | No | — | ISO 8601 datetime — only check-ins **before** this time |
| participantType | No | — | Filter by type (e.g., `VIP`, `Speaker`, `General`) |
| badgePrinted | No | — | `true` = only printed, `false` = only not printed |
| sort | No | desc | Sort by check-in time: `desc` (newest first) or `asc` (oldest first) |

**Example Requests:**

```
# First page, default 100 records, newest first
GET /api/v1/inbound/attended?eventCode=EVT-2026-ANNUAL

# Page 2
GET /api/v1/inbound/attended?eventCode=EVT-2026-ANNUAL&limit=100&offset=100

# VIPs checked in during morning session (9am–12pm window)
GET /api/v1/inbound/attended?eventCode=EVT-2026-ANNUAL&participantType=VIP&since=2026-06-15T09:00:00Z&before=2026-06-15T12:00:00Z

# Incremental pull — only new check-ins since last sync
GET /api/v1/inbound/attended?eventCode=EVT-2026-ANNUAL&since=2026-06-15T14:30:00Z&sort=asc

# Who checked in but didn't print a badge?
GET /api/v1/inbound/attended?eventCode=EVT-2026-ANNUAL&badgePrinted=false

# Max page size, oldest first (for bulk export)
GET /api/v1/inbound/attended?eventCode=EVT-2026-ANNUAL&limit=500&sort=asc
```

## Error Responses

All endpoints return consistent error formats:

| Status | Condition | Response |
|--------|-----------|----------|
| 400 | Validation failed (bad/missing fields) | `{ "error": "Validation failed", "details": { "fieldName": ["message"] } }` |
| 401 | Missing or invalid API key | `{ "error": "Missing or invalid API key. Expected: Authorization: Bearer grt_..." }` |
| 401 | Revoked API key | `{ "error": "Invalid or revoked API key" }` |
| 403 | IP not in allowlist | `{ "error": "IP address not allowed for this API key" }` |
| 403 | Feature not enabled | `{ "error": "Inbound API is not enabled for this account" }` |
| 404 | Event not found | `{ "error": "Event not found for code: EVT-XXX" }` |
| 429 | Rate limit exceeded | `{ "error": "Rate limit exceeded. Please slow down." }` |
| 500 | Server error | `{ "error": "Failed to process ..." }` |

## API Key Management

API keys are managed through the Greet admin UI (super admin only) or via authenticated admin endpoints:

| Endpoint | Method | Description |
|----------|--------|-------------|
| `/api/customers/:customerId/api-keys` | GET | List all API keys for a customer |
| `/api/customers/:customerId/api-keys` | POST | Generate a new key (body: `{ "name": "..." }`) |
| `/api/customers/:customerId/api-keys/:keyId` | DELETE | Revoke a key (soft delete) |

The raw key (`grt_...`) is returned **only once** on creation. It is stored as a SHA-256 hash — if lost, generate a new one.

## Security Summary

| Control | Implementation |
|---------|---------------|
| Authentication | Bearer token (SHA-256 hashed at rest) |
| Authorization | Key scoped to customer; feature flag gated |
| IP Restriction | Optional allowlist per key |
| Rate Limiting | 60 req/min per key |
| Payload Limit | 1 MB max |
| Input Sanitization | All string inputs HTML-sanitized |
| Audit Trail | Every request logged with action, key ID, IP, and payload summary |
| Safe Updates | Attendee check-in, badge, and workflow data never overwritten |

## Example: Full Sync Flow (cURL)

```bash
# 1. Push event
curl -X POST https://greet.certain.com/api/v1/inbound/events \
  -H "Authorization: Bearer grt_your_key_here" \
  -H "Content-Type: application/json" \
  -d '{
    "eventCode": "EVT-2026-ANNUAL",
    "name": "Annual Conference 2026",
    "eventDate": "2026-06-15T09:00:00Z",
    "timezone": "America/Los_Angeles"
  }'

# 2. Push attendees
curl -X POST https://greet.certain.com/api/v1/inbound/attendees \
  -H "Authorization: Bearer grt_your_key_here" \
  -H "Content-Type: application/json" \
  -d '{
    "eventCode": "EVT-2026-ANNUAL",
    "attendees": [
      {
        "externalId": "REG-10042",
        "firstName": "Jane",
        "lastName": "Smith",
        "email": "jane@example.com",
        "participantType": "VIP"
      }
    ]
  }'

# 3. Push sessions
curl -X POST https://greet.certain.com/api/v1/inbound/sessions \
  -H "Authorization: Bearer grt_your_key_here" \
  -H "Content-Type: application/json" \
  -d '{
    "eventCode": "EVT-2026-ANNUAL",
    "sessions": [
      {
        "externalId": "SESS-301",
        "name": "Keynote",
        "startTime": "2026-06-15T09:00:00Z",
        "endTime": "2026-06-15T10:30:00Z",
        "capacity": 500
      }
    ]
  }'

# 4. Pull attended list (after event)
curl -s "https://greet.certain.com/api/v1/inbound/attended?eventCode=EVT-2026-ANNUAL" \
  -H "Authorization: Bearer grt_your_key_here" | jq .

# 5. Push session registrations
curl -X POST https://greet.certain.com/api/v1/inbound/session-registrations \
  -H "Authorization: Bearer grt_your_key_here" \
  -H "Content-Type: application/json" \
  -d '{
    "eventCode": "EVT-2026-ANNUAL",
    "registrations": [
      {
        "attendeeExternalId": "REG-10042",
        "sessionExternalId": "SESS-301",
        "status": "registered"
      }
    ]
  }'
```
